## Critical Rules

### Rule 1: Heads Down Entry (Ctrl+N) is the DEFAULT navigation mode on 1040 — on EVERY screen, no exceptions
HDE is not a fallback for stuck checkboxes — it is the primary way data goes into a 1040 return. Sood, Dvorak, and Link all lost significant time (60 min, 30 min, 20 min respectively) the moment direct clicking was attempted on a multi-checkbox screen. Hammoud re-confirmed the pattern — any screen entered without Ctrl+N first becomes a pixel-hunt.

**THE RULE, HARDENED (2026-04-11, post-Hammoud):** `Ctrl+N` is your **FIRST action on every screen**, without exception. Not "screens with more than a handful of fields." Not "checkbox-heavy screens." **Every** screen. Even a screen with three fields benefits because HDE gives you field-number navigation that never misclicks. Even a screen you've already seen benefits because you stay in keyboard mode and don't break rhythm. The old "more than a handful of fields" carve-out is deleted — it encouraged me to skip Ctrl+N on "simple" screens, then mouse-hunt anyway.

**Flow for every single screen:**
1. Land on screen (via search box `W2`, `99I`, `99R`, etc.)
2. `Ctrl+N` — FIRST action, always
3. ONE screenshot to read the field-number overlay (only the first time you've ever seen this screen — after that, use the atlas below)
4. ONE `computer_batch` call that plays out the entire screen from the pre-processed data
5. `Ctrl+N` to exit HDE, `Esc` to exit the screen
6. That's 3 tool calls for the whole screen — screenshot, batch, (optional) verification screenshot

**Pre-processing as a force-multiplier:** Preprocessing (Phase 1) hands you a structured JSON of every value that needs to be typed — payer names, addresses, box amounts, SSNs, DOBs, ZIPs, etc. By the time you land on a data entry screen, there is **ZERO extraction work to do at entry time**. You already know what field 6 on 99R is going to be. You just need to type it. Entry should feel purely mechanical — if you're re-reading a PDF or workpaper mid-entry to find a value, preprocessing was incomplete and the time-leak is in Phase 1, not Phase 2. Fix Phase 1 next time rather than slowing down Phase 2.

**Atlas > rediscovery:** Drake's HDE field numbers are **static per screen type**. W-2 field 6 is always Social Security wages regardless of return. 99I field 2 is always Box 1 interest. Capturing the field-number overlay once and persisting it to the atlas below (Rule 27) means every subsequent W-2 gets entered in seconds from memory + preprocessing, not from another round of pixel-hunting the overlay.

Screens where HDE is mandatory (never use direct clicks here):
- Form 8867 Due Diligence — all 12+ checkboxes plus the 5-question bottom section
- Form 8863 Education Credits — the AOTC and 1098-T checkboxes
- Dependent Due Diligence tab — Yes/No/N/A columns are ~25px apart and misclick constantly
- Form 5329 — TS code + QTP/ESA/IRA sections
- IDS screen — "Taxpayer did not provide" / "Spouse did not provide" checkboxes
- Direct Deposit (DD) — field 13 for the Checking repeat checkbox
- Schedule B — ~20 yes/no interest/dividend questions
- K-1 QBI summary — MFC column on every row

Screens where HDE is strongly preferred (use it unless there is a specific reason not to):
- Screen 1 / Screen 2 — taxpayer and spouse info (address, DOB, occupation all batch cleanly)
- Dependents tab — name, SSN, relationship, DOB all numbered
- Schedule C — all expense lines (skip zero lines by not navigating to them)
- Schedule E — rental address, income, expense lines
- 99N, 99D, 99I — payer info and box amounts
- PIN — ERO PIN, taxpayer PIN, spouse PIN, phone

Screens where direct clicks still make sense:
- Opening a detail worksheet by double-clicking an amount field (HDE won't open sub-windows)
- Clicking the States tab / K-1 partner tabs / toolbar buttons
- Selecting from an already-open dropdown list
- Navigating between screens via the search bar at top-left

**The HDE entry pattern (VERIFIED on Shah Form 8867 — 11 checkboxes in one batch):**
- For **checkboxes**: `type "<field#>"` → `key "Return"` → `key "space"` → `key "Return"`. That's 4 actions per checkbox, no clicks required — the HDE dialog is already focused after Ctrl+N. Chain all the checkboxes into a single `computer_batch` call.
- For **text/amount/date fields**: `type "<field#>"` → `key "Return"` → `type "<value>"` → `key "Return"`. Same 4-action pattern.
- After each inner `Return`, the HDE dialog returns to the "enter desired field number" state, so the next field number can be typed immediately — no clicks, no re-focusing, no between-field screenshots.

One screenshot at the start of the screen (to get the field-number map from the Ctrl+N reveal), one batch call for all entries, one screenshot at the end to verify. A full Form 8867 pass (typical MFJ with CTC: ~11 checkboxes) is **one** `computer_batch` call.

This rule alone would have saved ~110 minutes across the first three 1040 returns.

### Rule 2: Check K-1 QBI MFC after every K-1 import
When K-1s are exported from partnership/S-Corp returns to a 1040, the QBI (Qualified Business Income) entries are auto-created but the MFC (Multi-Form Code) field may be blank. After importing K-1s:
1. Navigate to the K-1 QBI summary screen
2. Check every row — if MFC is blank, set it to `1`
3. Drake shows a warning dialog about missing fields but saves anyway — don't ignore it

### Rule 3: Q7a on Form 8867 — Leave BLANK for most clients
Q7a asks about Form 8862 (recertification after prior EIC/CTC denial). Answering "Yes" tells Drake the client needs Form 8862, triggering EF error 5798. For most clients who were NOT previously denied these credits, leave Q7a blank. Do NOT set it to Yes or N/A.

### Rule 4: Batch all 8867 answers in one pass
Form 8867 has ~12 questions across two sections. Answer ALL of them in a single Heads Down Entry session before calculating. Iterating (answer one → calculate → find next error → answer → calculate) wastes 5 minutes per cycle.

### Rule 5: Fill ALL blue fields on first pass
Blue fields = required for e-filing. Missing them means EF errors that waste calculate-fix cycles. Common blue fields on 1040:

**Screen 1:** SSN, First Name, Last Name, Address, City, State, ZIP, Filing Status, DOB
**Screen 2:** Spouse SSN, First Name, Last Name, DOB (if MFJ)
**Dependents:** SSN, First Name, Last Name, Relationship, DOB, Months in home
**Schedule C:** Business code, Business name, Address, City, State, ZIP, Accounting method, EIN (if applicable)
**99N:** Payer name, Payer address, Amount, Link to Schedule C
**PIN:** ERO PIN, Taxpayer PIN, Phone number
**1099-DIV:** Payer name, Ordinary dividends, Qualified dividends

### Rule 6: 99N (1099-NEC) flows automatically to Schedule C — DO NOT double-count
When a 1099-NEC is entered on the **99N screen** and linked to a Schedule C, Drake automatically flows that 1099-NEC amount into Schedule C gross receipts. If the client's P&L total gross income INCLUDES the 1099-NEC amount (which it almost always does), then Schedule C line 1 "Gross receipts and sales" must contain ONLY the non-1099 portion:

**Formula:** Schedule C Line 1 = Total P&L Gross Income − Sum of all linked 1099-NEC amounts

**Example (Dvorak):**
- Total P&L Gross Income: $138,832
- 1099-NEC from First American Title: $103,918 (on 99N screen, linked to Schedule C)
- Schedule C Line 1 should be: $138,832 − $103,918 = **$34,914**
- Drake then calculates: $34,914 (manual) + $103,918 (99N flow) = $138,832 ✓

**What goes wrong if you enter the full P&L amount:** Drake shows $138,832 + $103,918 = $242,750 in gross receipts — nearly double the correct income. This inflates total income, SE tax, and balance due dramatically.

### Rule 7: Fill Dependent Due Diligence tab on FIRST pass
The dependent screen has TWO tabs: "Dependent" (basic info) and "Due Diligence" (required for CTC/ODC). Both must be filled on the first visit — don't wait to discover it from an EF error. See the Dependent Due Diligence section below for what to answer.

### Rule 8: Form 8867 Q7 must be "Yes"
Q7 asks "Did you ask the taxpayer if any credits were disallowed or reduced in a previous year?" — the answer is always **Yes** (because we asked). "No" means we failed to ask, which triggers EF error 4906. This is different from Q7a (which should be BLANK).

### Rule 9: Always review prior year return thoroughly
Cross-reference the prior year (PY) return for: estimated payments, depreciation schedules, NOL carryforwards, Schedule E rental properties, recurring income sources. The PY return is the best checklist for what the current year return needs.

### Rule 10: Review ALL pages of ALL uploaded documents
Never skip pages regardless of file size. Tax documents can have critical data on any page — 1099s, K-1s, or supporting schedules can appear anywhere in the workpaper.

### Rule 11: Direct Deposit "Repeat account information" is MANDATORY
The DD screen has TWO rows of routing/account fields. Drake does NOT auto-copy from row 1 to row 2 — you must manually retype the routing and account numbers into the Repeat row, AND re-check the Checking/Savings checkbox (HDE field 13 for Checking repeat). Missing this triggers EF error 5244.

**Patel update (2026-04-11): field 11 "Repeat RTN" is the #1 forgotten field.** If field 12 "Repeat Account" is populated but field 11 is blank, Drake still fires EF 5244 ("Direct Deposit information was duplicated incorrectly"). Always fill all four fields in the same HDE batch: 9 (RTN), 10 (Account), 11 (Repeat RTN), 12 (Repeat Account).

**Patel update: DD field 1 (Federal Selection) depends on federal refund vs balance due.** Use `Y` only if there's a federal refund to direct-deposit. If federal is balance due (as on Patel), set field 1 to `N` — otherwise Drake tries to direct-deposit a refund that doesn't exist. The DD screen should still be populated if there's a STATE refund being direct-deposited.

**Patel update: DD field 2 (State/City Selection) is a state-code dropdown, NOT a Y/N field.** Earlier rules said "type A". For a state refund being direct-deposited, enter the 2-letter state code (e.g., `CA`). Typing `Y` triggers "Your entry is not VALID for the current Field! State/City Selection (drop list)".

**DD screen Tab-order flow (VERIFIED on Shah return — use this instead of click-hunting):**
1. Federal selection dropdown — type `Y` (if federal refund) or `N` (if federal balance due — Patel 2025)
2. Tab → State/city selection dropdown — type the 2-letter state code (e.g., `CA`, `IL`, `MO`) or `A` for all eligible states
3. Tab → 709 checkbox (skip)
4. Tab → Federal deposit amount (leave blank = full refund)
5. Tab → State deposit amount (leave blank = full refund)
6. Tab → **RTN** (type routing number; the "Name of financial institution" field is NOT in the tab order and is optional)
7. Tab → Account number (type account number)
8. Tab → Checking checkbox → Space to check
9. Tab → Savings checkbox (skip with Tab)
10. Tab → RTN (repeat) — type routing number again
11. Tab → Account (repeat) — type account number again
12. Tab → Checking (repeat) → Space to check
13. Close screen with the X button on the Data Entry title bar (safe save-and-exit)

**DD screen trap:** The Federal Selection dropdown is the first field and has focus when the screen loads. If your click lands anywhere before the RTN field, typed text pipes into that dropdown and triggers a modal validation dialog ("Your entry is not VALID for the current Field! Federal Selection (direct entry) <garbage>"). To recover: click the X on the dialog title bar, press Delete to clear the dropdown, type `Y`, then Tab forward through the fields. Do NOT attempt pixel clicks on the DD row of input boxes — Tab navigation from the top field is the only reliable path.

### Rule 12: IDS screen — use direct click, NOT search box
Typing "ID" in Drake's screen search navigates to the Idaho state package, not the IDS (Identification) screen. Click IDS directly from the General tab's "Electronic Filing and Banking" section (below BANK, above USE). For clients who don't provide driver's license, check BOTH "Taxpayer did not provide" AND (if MFJ) "Spouse did not provide" checkboxes. Missing this triggers EF error 500.

### Rule 13: Form 5329 has a required TS code
When QTP/ESA/IRA data auto-populates on Form 5329 (e.g., from 1098-T student distributions), the TS (Taxpayer/Spouse) code field at the top is BLUE/required but easy to miss. Always enter T or S on first pass. Missing this triggers EF REQUIRED error for screen 5329. Prior-year carryover "State Info" code can usually be ignored.

### Rule 14: Two 1098-T forms for the same student — combine into one 8863 entry
When a student attends two schools in the same year (e.g., transfer), combine Box 1 amounts into a single Form 8863 entry using the primary institution's name, EIN, and address. Do not create two separate 8863 screens — Drake's AOTC calculation is per-student, not per-institution.

### Rule 15: Form 8867 "5 questions below Q15" is actually 5, not 4
The form description says "four questions below question 15" but there are actually FIVE: (a) nonresident alien, (b) qualifying child of another person, (c) taxpayer's main home in US, (d) spouse's main home in US [MFJ only], (e) eligible as dependent of another. The MFJ spouse question is the "extra" one. All must be answered.

### Rule 16: PIN screen field 2 (ERO's PIN signature) — direct click EXCEPTION to HDE
The PIN screen's "ERO's PIN signature" field is a password-masked field that shows 5 dots regardless of what's typed. **HDE entry does NOT reliably persist this field** — on Patel 2025 the value was entered via HDE three separate times (with the dots rendering each time), and every time the field was blank after exiting the screen, triggering EF 5084/672 "MISSING PIN: No preparer's PIN has been entered on the PIN screen." **Re-confirmed on Hammoud 2025** — same symptom, same fix. This bug is now verified on **two consecutive returns** (Patel + Hammoud). Treat it as a permanent Drake Tax 2025 bug, not a one-off.

**Workaround:** direct-click field 2 (coordinate x=718, y=184 on Drake Tax 2025 PIN screen — verified Hammoud), type `75757`, press Tab. This is the only HDE exception for 1040 PIN entry that has been verified. All other PIN screen fields (Taxpayer PIN, Spouse PIN, Date, etc.) can still be entered via HDE.

**Non-negotiable pre-flight:** Before starting ANY 1040 return, grep this file for "field 2" and re-read this rule. The user has flagged this as the #1 recurring skill-discipline failure. HDE is the default everywhere else — field 2 on PIN is the exception, and forgetting it costs one full EF-error-fix-recalc cycle every time.

If the user is preparing a state return that also needs its own ERO PIN (IL 9074, TX 9074), verify field 2 persisted BEFORE calculating — open the PIN screen, check for 5 dots next to "ERO's PIN signature", exit. Federal PIN failure cascades to state PIN failures.

### Rule 17: CA 540NR — Schedule CA NR line 7 is REQUIRED
Even when NR screen line 5 "Nonresident ALL year" is set to the home state (e.g., TX for both TP and Spouse), Drake still requires an explicit Yes/No answer on line 7 "Owned a home/property within CA" for both Taxpayer and Spouse. Blank = EF CA 1392. For TX residents with only CA-source wages (the common SpaceX / Big Tech case), answer **No** for both.

**HDE field numbers on NR screen (verified on Patel 2025):**
- Line 7 Taxpayer Yes = 30
- Line 7 Taxpayer No = **31** ← set this
- Line 7 Spouse Yes = 32
- Line 7 Spouse No = **33** ← set this

Pattern per field: `type "31" → Return → space → Return`, then `type "33" → Return → space → Return` — chain into one computer_batch.

### Rule 18: CA HCM screen — check field 1 "YES EVERYBODY had MEC" on first pass
For Lineal clients with full-year employer health coverage (1095-C from SpaceX, Amazon, Microsoft, etc.), the correct answer on the California HCM (General Health Coverage Information) screen is to check **field 1** "YES - EVERYBODY on this return had minimum essential coverage for every month of 2025". Missing this triggers EF CA 6205 "CA3853 - Verification of MAGI required" because Drake thinks the return needs Form CA 3853.

**Do NOT check field 4** "I have verified the MAGI for all household members on the CA3853 form" unless CA3853 is actually being filed (Marketplace coverage or exemptions claimed). Field 1 is the cleaner fix for the employer-coverage case because it eliminates CA3853 entirely.

**HDE pattern:** `type "1" → Return → space → Return` on the HCM screen.

### Rule 19: CA errors hide in CAMSG, not in the calc-dialog EF Messages list
**Dangerous false-positive trap:** The calculation-results dialog can show "CA540NR Eligible For E.F." with a GREEN checkmark even when the California return has EF errors. Those errors live in the **CAMSG** (California Messages) page inside View/Print, NOT in the main EF Messages list. The green checkmark only means the state package was produced, not that it's error-free.

**Always verify CA returns via View/Print:**
1. Click View/Print after Calculate
2. Scan the form tree in the left panel for red items — look specifically for `CAMSG` (or `MESSAGES` for federal, `ILMSG` for IL, etc.) in the California section near the top
3. Red = errors; absence = clean
4. Right-click any red message → "View Full Text" for the detail

CA 1392 (NR line 7) and CA 6205 (CA3853 MAGI) both surfaced this way on Patel. Both are easy to miss if you trust the green checkmark.

### Rule 20: Exit button coordinate — click x=985 to hit the icon, not the dropdown arrow
The Data Entry title bar's Exit button has a dropdown arrow at x≈987-990 that opens an "Exit Without Saving" menu. To safely exit (save and close), click at **x=985, y=122** to hit the icon itself. Clicking further right catches the dropdown and can lose work if "Exit Without Saving" is accidentally selected.

**However, prefer `Esc` over clicking Exit — see Rule 21.**

### Rule 23: 8949 — Sold=VARIOUS and S/L=blank both trigger EF 5310 (Hammoud 2025)
The 8949 schedule (via the D/8949 screen) has two adjacent traps that both surface as EF Federal error **5310 "INVALID DATE"**:

**Trap A — Sold date "VARIOUS" is rejected by e-file.** When entering a wash-sale summary row or any rolled-up ST/LT summary, it is tempting to type `VARIOUS` in both Acquired and Sold. Drake accepts this on entry but the e-file schema rejects it on calculation: `Date Sold must be a valid date or 'VARIOUS'`. The message is misleading — in practice, Sold MUST be a specific date. For calendar-year 2025 summaries, use **`12-31-2025`** as the Sold date. Acquired can still be `VARIOUS`.

**Trap B — Acquired=VARIOUS requires an explicit S/L code.** When Acquired is `VARIOUS`, Drake cannot auto-determine short-term vs long-term, so the **S/L column is mandatory**. A blank S/L column with Acquired=VARIOUS fires EF 5310 even though the error text talks about dates. Enter `S` or `L` in the S/L column on every summary row where Acquired=VARIOUS.

**Hammoud pattern (M1 — wash-sale summary, 4 rows):**
- Row 1: JPM ST summary — Acquired=VARIOUS, Sold=`12-31-2025`, S/L=`S`
- Row 2: JPM LT IBIT — Acquired=VARIOUS, Sold=`12-31-2025`, S/L=`L`
- Row 3: Pershing ST summary — Acquired=VARIOUS, Sold=`12-31-2025`, S/L=`S`
- Row 4: Pershing LT summary — Acquired=VARIOUS, Sold=`12-31-2025`, S/L=`L`

**Field 5 on the 8949 screen is the 1099-B box code (numeric, not letter):**
- `1` = Basis reported to IRS (Box A short-term / Box D long-term)
- `2` = Basis NOT reported to IRS (Box B short-term / Box E long-term)
- `3` = Not reported on 1099-B (Box C short-term / Box F long-term)

Drake's on-screen help shows A/B/C/D/E/F but the field only accepts the numeric code. Typing `A` or `E` returns "Your entry is not VALID for the current Field!" On Hammoud: JPM and Pershing both issued 1099-Bs with basis reported, so all four summary rows got field 5 = `1`.

### Rule 24: 1099-R screen recipient-override fields 16-21 can carry over from another return — ALWAYS clear them on first entry
The 1099-R screen has override fields 16-21 for recipient name/address when the recipient isn't the primary taxpayer. On Hammoud, these fields auto-populated with **a completely different client's name and address** (believed to be stale data from a prior session). If you don't notice and clear them, Drake prints the 1099-R with the wrong recipient name, which flows into the federal return and can trigger IRS matching failures.

**Mandatory first-pass check:** After landing on a fresh 1099-R screen (via `99R` in search), screenshot and look at fields 16-21 BEFORE entering any data. If ANYTHING is populated there — even if it looks like the current taxpayer — blank all six fields explicitly. HDE pattern per field: `type "<16-21>" → Return → Home → Shift+End → Delete → Return`. Six fields, one batch.

This is a stale-state bug in Drake Tax 2025 and probably applies to other recipient-override screens (1099-MISC, 1099-NEC). Treat recipient-override fields as "guilty until proven innocent" on every new-entry screen.

### Rule 25: Student loan interest auto-disallowed above MFJ MAGI phaseout ($200k for 2025) — skip the entry
MFJ student loan interest deduction phases out between **$170,000 and $200,000** of modified AGI for 2025. Above $200k, the deduction is zero regardless of what's entered. Hammoud AGI was ~$397k, so the 1098-E workpaper entry would have been a pure waste of a screen.

**Pre-flight rule:** Before entering 1098-E interest, estimate current-year MAGI from W-2 box 1 + other income. If MFJ MAGI clearly exceeds $200k (single $95k), SKIP the 1098-E screen entirely. Note in the client file "SL interest not entered — above phaseout." Same rule applies to IRA deduction ($143k MFJ with retirement plan), Roth contribution ($246k MFJ 2025), and tuition and fees credits.

**Single-threshold quick reference (2025 MFJ):**
| Deduction / Credit | Phaseout Start | Phaseout End |
|---|---|---|
| Student loan interest | $170,000 | $200,000 |
| Traditional IRA (active participant) | $126,000 | $146,000 |
| Roth IRA contribution | $236,000 | $246,000 |
| Lifetime Learning Credit | $160,000 | $180,000 |
| AOTC | $160,000 | $180,000 |
| Child Tax Credit | $400,000 | per-$1,000 phase-in |

### Rule 26: ZIP-to-city auto-lookup can "correct" a valid city to a USPS preferred name — accept it
Drake uses USPS ZIP data to auto-populate city when you type a ZIP. For some ZIPs the USPS "preferred name" differs from the common city name — e.g., ZIP 41015 → "Latonia" (not "Covington", which is the common name residents use). On Hammoud the user's 1099-R payer address was Covington KY 41015 and Drake rewrote city to Latonia after ZIP entry.

**Rule:** Accept Drake's city lookup. Do NOT overwrite it back to the common name — that triggers a validation warning and Drake will just rewrite on the next Tab. If the lookup produces a wrong STATE (rare), clear the ZIP, retype slowly, and let Tab advance normally.

### Rule 27: HDE Atlas discipline — capture every new screen once, reuse forever
Drake HDE field numbers are **static per screen type**. Form W-2 field 6 is "Social Security wages" on Sood's W-2, Patel's W-2, Hammoud's W-2 if he'd had one, and every W-2 in every future 1040. This means a captured field map is a **permanent asset** — and the fact that this skill still lacks a W-2 atlas after 7 returns is a pure discipline failure.

**The mandatory capture flow for any screen you haven't documented yet:**
1. Navigate to the screen (via search box)
2. `Ctrl+N` — first action always (Rule 1)
3. ONE screenshot with the field-number overlay visible
4. **Before typing a single value**, transcribe the entire overlay into the "HDE Field Number Atlas" section of this file — field number → field label, for every numbered field on the screen. It takes 30 seconds. Do it.
5. THEN enter the client's data using the atlas entries you just wrote
6. Commit the atlas update to GitHub at Phase 4 like any other learning

**The mandatory reuse flow for any screen you've already documented:**
1. Navigate to screen
2. `Ctrl+N` — first action always
3. No screenshot. Go straight to `computer_batch` using the atlas field numbers + the preprocessed values. This is where return-5-in-15-minutes comes from — no screenshot latency between screens, just batch-after-batch blasts at the keyboard.
4. ONE verification screenshot only if something feels off after exit

**Atlas update timing:** Capture happens at Phase 2 entry time, committed to the 1040.md working copy immediately; the atlas is pushed at Phase 4 along with the rest of the return's learnings. If you open a new screen type and don't capture it, that's a Phase 4 audit finding (same severity as forgetting HDE on a screen).

### Rule 28: Preprocessing is the single source of truth for Phase 2 values
By the time you land on a data entry screen, you should already have a structured dict of `{field_number: value}` pairs for every field on that screen — name, SSN, DOB, address, ZIP, W-2 box amounts, 1099 box amounts, all of it. Preprocessing (Phase 1, via `scripts/preprocess.py` or manual extraction) is where that dict gets built. **Phase 2 is pure mechanical typing from the dict — zero extraction, zero lookup, zero PDF re-reading.**

**Smell test:** if during a return you catch yourself opening the workpaper PDF or the prior-year return to read a value, STOP. That's a Phase-1 failure, and the cost of re-opening + scanning + finding + typing is ~30-60 seconds per field. Multiply that by 40 fields on a typical return and you've burned 20-40 minutes on a step that preprocessing was supposed to eliminate.

**What "complete preprocessing" looks like for a 1040:** a JSON or markdown file with sections per screen, each section containing the exact field numbers (from the atlas) and the exact string to type. Reading that file end-to-end should be enough context to enter the entire return without opening any other document. If it isn't, keep extending the preprocessor until it is. The one-time cost of building better preprocessing pays back on every subsequent return.

### Rule 22: Screenshot discipline inside a screen
`computer_batch` executes a whole sequence of actions in one model↔tool round trip; individual tool calls each cost a full round trip (~3 sec each). The mandatory pattern for every data entry screen is:

1. **ONE** screenshot at the start of the screen (to read the field-number overlay)
2. **ONE** `computer_batch` call containing every HDE action for that screen (every `<field#> → Return → <value> → Return` sequence chained together)
3. **ONE** verification screenshot at the end of the batch

Do NOT take a screenshot between fields. Do NOT take a screenshot after each HDE action. Do NOT `left_click` and then immediately `screenshot` to "see what happened" — put both actions inside the same `computer_batch`. A full Form 8867 pass (~12 checkboxes) should be exactly 3 tool calls total: screenshot, batch, screenshot. If you're making more than 3 tool calls per data entry screen, you are leaking time.

## K-1 QBI — Detailed Notes

When K-1s are imported from pass-through entities, Drake creates QBI entries on the K-1 QBI summary screen. Key fields:

| Field | Description | Required? |
|-------|-------------|-----------|
| Source | K1P (partnership) or K1S (S-Corp) | Auto |
| EIN | Entity EIN | Auto |
| Entity Name | Partnership/S-Corp name | Auto |
| MFC | Multi-Form Code | **YES — often blank, must set to 1** |
| QBI Amount | Qualified business income | Auto from K-1 |
| W-2 Wages | Entity W-2 wages | Auto from K-1 |
| UBIA | Unadjusted basis of assets | Auto from K-1 |

**The MFC field is the #1 issue.** Drake shows a dialog warning that fields are missing but still saves. Always verify MFC after K-1 import.

## Speed Improvement Checklist

Each return should be faster than the last. The main time sinks and their fixes:

| Time Sink | Typical Minutes Lost | Fix |
|-----------|---------------------|-----|
| Pixel-hunting for field coordinates | 10-30 min | Turn on HDE at the top of every screen (Rule 1) — coordinates are replaced by field numbers |
| Misclicking adjacent Yes/No/N/A columns | 5-15 min | HDE navigates by field number, cannot hit the wrong column |
| Missing blue fields → EF error loop | 15-30 min | Fill ALL blue fields on first pass in HDE (Rule 5) |
| 8867 checkbox clicking without HDE | 20-60 min | HDE is mandatory on 8867 — use the field-number map above |
| 8863 / 5329 / IDS checkbox clicking without HDE | 10-30 min | HDE is mandatory on these screens too (Rule 1) |
| 99N double-counting Schedule C | 30-60 min | Subtract 99N from P&L total (Rule 6) |
| Forgetting dependent due diligence | 15-30 min | Fill both tabs on first visit in HDE (Rule 7) |
| Iterating calculate → fix → calculate | 10-30 min | Fix every known-blue field in one HDE sweep, then calculate once |
| Not reading PY return | 30-60 min | Always cross-reference PY (Rule 9) |
| Screenshot-per-field overhead | 10-20 min | One screenshot at start of screen, one at end — build HDE batch from the start image |
| Context window exhaustion | 30-60 min | Batch more, screenshot less, HDE not click-hunt |

**After every return:** Update this file with new coordinate data, new EF errors, and new batch sequences discovered. Push to GitHub. The goal is that return #5 takes under 15 minutes.
